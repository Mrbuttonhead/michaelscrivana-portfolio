Case 01 / 05

DartReviewerPX — catching the diff before legal does.

An agentic pre-review for regulated packaging artwork. One missed character in this category can mean a recall, so the design problem was trust at a level higher than convenience.

# dartreviewerpx
Role
Designer · builder · prompt architect
Stack
Claude Opus on Azure · React · PDF.js · Node
Year
April–ongoing 2026
Surface
Graphics Operations · ~30 reviewers

The problem

Pharmaceutical packaging artwork passes through a Legal, Medical, Regulatory (LMR) review before it ships. The reviewers are good. The reviewers are slow. And the cost of a single uncaught diff — an inverted asterisk, a missing trademark, a one-character claim drift — is a recall, with a cost shape that nobody on the design side wants to live with.

I sat with the graphics operations team and watched what they did between rounds. The pattern was hand-eyeballing a fifteen-page PDF against a slightly different fifteen-page PDF, line by line, often at the end of a long day. That was the wedge: an agent could be the first reviewer, surface the diffs, and let the human do the judgement work on a much smaller set.

The design approach

The temptation with a vision model is to throw both PDFs at it and ask "what's different." That fails fast for a reason most demos hide: image resizing on the server side erases single-character drift before the model ever sees the page. I learned this the hard way. So the architecture became dual-path: extract text via pdfjs on the client, render each page as an image, and pass both to Claude Opus together. Text catches the asterisks and trademarks. Vision catches the layout drift and the color shifts. The model reconciles.

The interface design was the other half. Reviewers don't want a wall of "differences." They want a triage queue, sorted by what could ship and what couldn't. So the output is a structured report with three layers — a one-line headline diff, a categorized table (brand, claim, regulatory, layout, typography), and a per-page deep view with the model's reasoning beside the visual.

What I built

  • A React + Node web app on Azure Static Web Apps with Entra-tenant SSO — Bayer-only, no public access.
  • A dual-pass review pipeline (text + vision) calling Claude Opus on the Anthropic passthrough endpoint, with structured JSON output and the raw reasoning preserved for audit.
  • A reviewer-facing diff view with severity sorting, per-page evidence, and an "accept / dismiss" loop that I use as the training signal for the next iteration.
  • A workshop demo at Bayer's Graphics Operations AI Workshop — live, with the team's real artwork — that produced the four converged friction patterns now feeding our PX roadmap.

What it changed

The point isn't speed. It's that the design surface made the agent legible enough that reviewers actually trusted it as a first pass. The "training signal" question — how do we know the agent is getting better? — is the next design question, and one I'd want to be working on at Anthropic. This is a representative case for the kind of quality-and-trust loop that AI-native tools need to solve before they can move from demo to default.

What I'd do next

Co-deploy with the LMR reviewers. Build the disagreement view — show me the diffs where the human and the agent disagreed last week, sorted by category. That's where the model gets sharper and where the reviewers get faster, in the same loop.

A single missed character in regulated artwork means a recall. The design surface had to make the agent legible enough to actually be trusted as a first pass.

Case 02 / 05

CreativeLab — one SKU, five channels, sixty seconds.

A node-based creative iteration platform for retailer-facing packaging variants. Designers compose pipelines on a React Flow graph instead of writing prompts.

# creativelab
Role
Designer · builder · orchestrator
Stack
React Flow · Node · Gemini NB2 (pivoting to Flux / gpt-image-2)
Year
May 2026, ongoing
Surface
Brand Design + Ecommerce · OAD Prime Day pilot

The problem

Retailer-facing packaging is a multiplication problem. One SKU shows up as a carton render on the brand site, a hero on Amazon, a tile on a retailer.com page, a vertical for social, a 6-second cut for digital out-of-home. Each surface has different ratios, different compliance constraints, different regulatory copy. Most teams handle this by sending the agency a brief and waiting a week. That cost shape kills experimentation.

The Ecommerce team came to me with a concrete need: prove out an OAD SKU across five channels with three regulated tweaks — a pending claim addition, a regulatory wording shift, an imagery refresh — in time for Amazon Prime Day. I had a week.

The design approach

The wrong move would have been to build a prompt UI. Designers don't think in prompts; they think in steps. A reference image goes in, a transformation happens, a check happens, an output comes out. So I borrowed the visual grammar of node editors — React Flow — and built the tool around composition, not prompting. Each node is a designed primitive: a reference input, a brand-rules check, an image generator, a channel-fit transform, a compliance reviewer, a final output. The user wires them. The graph runs.

The aesthetic choice was deliberate. I leaned into the cool, technical feel of node-graph tools (think Houdini, Blender's compositor, Figma's auto-layout) rather than the soft pastel feel of consumer AI tools. The audience — brand designers and ecommerce ops — respects that grammar. It signals "you are in control of the steps," which is the actual product promise.

What I built

  • A React Flow graph editor wired to swappable image-generation backends (Gemini Nano Banana 2 today, with the MGA img_generator workflow and Foundry gpt-image-2 / Flux planned as second-stage fidelity backends).
  • A brand-rules layer co-owned with the Brand Design sub-home, exposed to the graph as a check node and to other AI surfaces via the Knowledge MCP server (see case 05).
  • An OAD pilot dataset — one SKU, five channels, three regulated tweaks — that became the anchor demo for Bayer's Science Leadership Team.
  • A capture pipeline (Creative Library) that ingests public retailer imagery as reference assets, so the tool works on day one without waiting for the brand team to upload their catalog.

What it changed

The early result wasn't the time savings. It was the conversation. Showing brand designers a graph editor instead of a chat box reframed AI in their workflow from "the thing that writes prompts for me" to "the thing I orchestrate." That reframing is what I'd want to design more of: AI surfaces that respect the user's existing skill, not surfaces that flatten everything into chat.

What I'd do next

Make the graph saveable as a reusable template. The Ecommerce team's "OAD Prime Day" graph is the first one; it shouldn't be the last instance of itself. Templates plus parameterization is the path from "useful one time" to "operating in the background of every launch."

Designers don't think in prompts. They think in steps. The tool is the graph.
React Flow Node Gemini NB2 MGA Workflow Brand-rules MCP Vite
Case 03 / 05

AI Hub — one front door, one identity, five tools.

A monorepo platform that hosts every AI tool I ship to the organization. Tools graduate here from pilot so work compounds across teams instead of evaporating between projects.

# ai-hub
Role
Designer · builder · platform owner
Stack
Vite · React · Azure Static Web Apps · Entra SSO
Year
2026
Surface
~115 designers, researchers, packaging engineers

The problem

Internal AI tools have a graveyard problem. Someone ships a clever pilot. It lives on a hostname nobody remembers. The colleague who built it leaves. The tool dies. Six months later somebody builds the same thing again because they didn't know the first one existed. Multiply by a 115-person org and you get a folder of dead URLs and a culture that doesn't trust internal tooling.

I'd watched this play out across the previous year. So when I started shipping AI tools at scale, I gave myself a constraint: nothing ships as a one-off. Every new tool lives under one front door, behind one identity provider, with one consistent shell. The tool is a route, not a deployment.

The design approach

The landing page is the platform's main design surface. It doesn't sell — it indexes. A grid of cards, each card a tool with a one-line description, an owner, a "stage" (pilot, beta, in-production), and a use-count. The grid is the source of truth: if a tool isn't on the grid, it doesn't exist for the org.

The shell each tool inherits is opinionated and tiny. Same nav, same auth, same footer, same opinionated empty states. Tool designers don't decide whether to add a logout button. They decide what their tool actually does. That's the whole point of a shell — it removes the boring decisions so the interesting ones get more attention.

What I built

  • A Vite + React monorepo on Azure Static Web Apps with Entra-tenant SSO — one auth boundary, five routes.
  • An opinionated shell — nav, auth, error states, telemetry — that every new tool inherits as a workspace template.
  • Five tools currently mounted: AI Hub landing · Brief Translator (turns prose briefs into structured creative briefs) · Persona Generator (synthetic personas, co-built with Product Research) · Case Study Sharpener (rewrites internal case studies against brand benchmarks) · Knowledge MCP console.
  • A "graduation" rubric — before a pilot mounts to the hub, it has to clear governance, identity, and a one-page case study. That rubric is the soft contract that keeps the platform from becoming the graveyard it was designed to replace.

What it changed

Inside three months, the hub stopped being something I evangelized and started being something colleagues asked to be put on. That's the only adoption signal that matters for an internal platform. The pattern I'd want to bring to Anthropic: treat internal-tool design as product design, not as a side effect of someone's pilot.

What I'd do next

Surface usage and trust signals on the grid itself. Which tools are people actually opening? Which tools made it past first use? The grid currently shows what exists. The next version should show what's earning its place.

A tool is a route, not a deployment. One front door, one identity, one shell — or you build the graveyard you were trying to replace.
Monorepo Vite React Azure SWA Entra SSO 5 tools
Case 04 / 05

Daily Briefing Pipeline — the design loop I run my own work through.

A personal operating system on a public dashboard. Node and Claude-on-Azure synthesize my vault, recent recaps, and initiative state into a one-screen briefing at 7am every morning, scheduled by launchd.

# daily-briefing
Role
Designer · builder · sole user (for now)
Stack
Node · Claude on Azure · launchd · Obsidian vault
Year
2026, running daily
Surface
PX Dashboard · my morning

The problem

I run multiple workstreams across PX and the cross-functional AI working team. Each one has its own initiative note, its own running backlog, its own stakeholder context. Most mornings I used to spend twenty minutes paging through the vault trying to remember what mattered before the first meeting hit. That's not work. That's startup tax.

So I built the surface I wished existed. Not a generic productivity dashboard — a briefing engine that reads the same vault I write into and produces the one-screen version of "what should be on your mind this morning."

The design approach

The briefing has to be skim-grade. If I read more than the headlines, the design failed. So the format is fixed and minimal: a section called Themes today (three to five terse headers with one-line orientations), a section called Open threads (stakeholder threads I owe a reply to), a section called Initiative state (which projects moved, which slipped, which I haven't touched in seven days), and a single closing line called What's the first move. That last line is the design point. Most personal briefings end with a list. Mine ends with a decision.

The pipeline is deliberately boring: a launchd plist fires a Node script at 7am, the script walks the last 72 hours of the vault and the last 14 days of recaps, hands the materials to Claude on Azure with a tightly-scoped system prompt, writes the output to a markdown file in the dashboard's public folder, and a static dashboard renders it. There's no app. There's a file. The file is the product.

What I built

  • A launchd-scheduled Node pipeline that reads my Obsidian vault and synthesizes a daily briefing via Claude on Azure's Anthropic passthrough.
  • A "skim style" system prompt — theme heading plus ten-word bullets, never dense em-dash prose — that I iterated on by giving myself the briefing every morning for a month and noting where I bounced.
  • A "Generate now" fallback button on the dashboard for the inevitable mornings when launchd misses (laptop closed, on a plane, time-change weirdness).
  • A second pipeline that the /recap, /bod, and /eod slash commands feed into — so my own behavior produces the inputs for tomorrow's briefing without me writing summary prose by hand.

What it changed

The dashboard isn't really the product. The product is the practice — the design loop where I write into the vault all day, slash-command my own recap at end of day, and wake up to a synthesis the next morning. The loop is the artifact. The dashboard is just where it lives.

This is the case that maps most directly to the Claude Code role: I'm already living inside the "personal OS" pattern the team is moving toward. I have opinions about what works in it, what doesn't, and where the design surface is still under-built. (More in the meta essay.)

What I'd do next

Make it a team-shaped artifact, not just a personal one. Same shell, same skim grammar, but with the permeable-membrane question: how much of the personal briefing should be shareable as a sub-team digest without losing the intimacy that makes the personal version actually useful?

The product isn't the dashboard. The product is the loop. The dashboard is just where it lives.
Claude on Azure Node launchd Obsidian Slash commands 7am daily
Case 05 / 05

Knowledge MCP — single build, three AI surfaces.

An MCP server exposing canonical organizational standards — brand rules, taxonomy, sub-home standards — to three AI surfaces from a single deploy. Three tools, one source of truth.

# knowledge-mcp
Role
Designer · builder · protocol-thinker
Stack
MCP (Streamable HTTP, 2025-06-18) · FastMCP · Node
Year
v0.1 May 2026
Surface
MGA · M365 Copilot · local Claude Code

The problem

Bayer has three approved AI surfaces — an internal GenAI platform (MGA), Microsoft 365 Copilot, and local Claude Code. Every team that builds a knowledge integration ends up building it three times, once per surface, and the three implementations drift. The brand-rules JSON the agency reads is different from the brand-rules the Copilot agent quotes is different from the brand-rules my own Claude Code session loads. Three sources of "truth," none of them true.

MCP solves this at the protocol layer, not the integration layer. If the canonical standards live inside an MCP server and every surface consumes them through the protocol, the three drift problem becomes one server problem. That was the design bet.

The design approach

The interesting design question with an MCP server isn't the wire format. It's the tool grammar. What tools do you expose? What do you not? How do you make the tool names self-documenting so an agent reasons correctly about when to call them?

I started with three. find_canonical(topic) returns the canonical artifact for a topic, with a citation. search(query) returns a ranked list across the knowledge corpus. list_cells() enumerates the six sub-home knowledge cells and their domains. Three tools, three verbs, three different shapes of intent. An agent calling these doesn't need a tutorial — the names tell it what each does.

The other design choice was where the data lives. The MCP server reads from a monorepo of markdown files versioned in Git, not from a database. That choice does two things at once: it makes contribution feel like writing a doc, not maintaining a system, and it gives every answer the MCP returns a stable Git URL as its citation. The agent says "per the OAD logo line semantics doc, paragraph 3," and you can click the citation to the line in the markdown.

What I built

  • A FastMCP server (Streamable HTTP, MCP spec 2025-06-18) exposing the three tools above, packaged as a Bayer Terraform-deployable Lambda module.
  • A monorepo of canonical knowledge cells — Brand Design seeded as the v0.1 pilot, with eight cells covering OAD brand standards, the Claritin brand system, and PX-wide rules.
  • Registration into Bayer's MGA platform (as a custom tool), Microsoft 365 Copilot (as a declarative-agent action), and local Claude Code (via .mcp.json). Same server, three surfaces.
  • A grammar doc — the README of the server — that defines the contribution pattern for the next five sub-home cells.

What it changed

This is the case I'm proudest of from a design-leverage standpoint. The build is small; the consequence is large. One MCP server collapses three integration projects into one. Other PX teams now have a contribution pattern, not a custom build. And canonical standards become a primitive AI surfaces can call, not a PDF nobody reads.

It's also the case that most directly speaks to the Claude Code team's posture — "stay close to the models," "rethink the basics." MCP is one of the basics being rethought right now. I want to be in the room when the next ones are.

What I'd do next

Author the missing five sub-home cells with their respective leads as co-contributors. The technical work is done; the social work is the bottleneck. (That's a familiar shape for anyone shipping infrastructure inside an organization.)

One MCP server collapses three integration projects into one. The build is small; the consequence is large.
