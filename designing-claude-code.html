Meta · Designing Claude Code

I don't just use Claude Code. I live inside it.

Twelve months of customizing the environment — hooks, subagents, slash commands, MCPs, a vault, and a daily-briefing pipeline — reshaped my design practice. A short essay on what that taught me about AI-native product surfaces.

Written

01 · The problem

Most "AI features" are surfaces grafted onto pre-AI products.

Last month I gave a 30-minute talk to my colleagues in Nashville called "AI for Designers." The spine was five principles — context, knowledge, conversation, memory, workflow — that turn Claude from a chatbot into a working environment. I built the talk to teach the practice I'd been quietly developing for a year. By the time I gave it, I'd noticed something quieter underneath the principles: I wasn't using Claude Code anymore. I was designing inside it.

The pattern I was reacting to is everywhere now. A SaaS product ships an "AI assistant." It's a chat window in the corner. It reads your data, it answers questions, it sometimes drafts something. The model is good. The surface is the same surface the product had before there was a model behind it.

That works as a feature. It fails as a design philosophy. Because the interesting design problem isn't "what goes in the chat window." It's the layer underneath — the working environment the user actually lives in. Files, configs, hooks, shortcuts, the way one tool composes with the next. The substrate. When the substrate is designed for AI, the chat window becomes the least interesting thing in the room.

Claude Code is the clearest example I've used of a product that took the substrate question seriously. A CLI with hooks, slash commands, MCPs, and subagents is not "a chatbot in a terminal." It's a new interaction grammar — one where the user authors the environment as readily as they invoke it. That's the bet I think Anthropic is making, and it's the bet I've been making in my own building.

02 · The custom surfaces

A year inside Claude Code, building outward.

I didn't set out to build a personal operating system. I just kept noticing small frictions, and Claude Code kept making them cheap to fix. A year on, the customizations have composed into something I rely on every day. Here's the surface area I authored, in roughly the order I built it.

/bod
Morning kickoff skill. Reads today's daily note, surfaces yesterday's open items, asks which to start. Replaces a fifteen-minute manual triage.
/eod
End-of-day close-out. Audits today's note, captures unfinished items, rolls them into tomorrow. The pair that makes /bod work the next morning.
/recap
Session recap. Appends the session's Claude work to the Obsidian daily note. Used so often it became muscle memory; the daily note is now a continuous transcript of how I think with the model.
vault-synthesizer
Subagent. Reads across my two-hundred-plus initiative notes and synthesizes state on demand. The first time it ranked my "top three things to ship this week" with full vault context, the calculus of what I could ask the model to do shifted.
brief-reviewer
Subagent. Critiques drafts against stakeholder feedback and missing cross-references before they go to leadership. A standing second pair of eyes that never gets tired and remembers the last six months.
px-mcp-knowledge
Custom MCP server. Exposes our organizational standards — brand rules, taxonomy, sub-home conventions — to Claude Code, MGA, and Microsoft 365 Copilot from a single deploy. Three tools, three surfaces, one source of truth.
SessionStart hook
Hook. Prints today's daily note and the initiatives modified in the last forty-eight hours every time I open a session. Context-as-greeting; I never start cold.
Stop hook
Hook. Text-to-speech via macOS say, toggled by a file flag. I can hear Claude's responses while making coffee. Small, but it changes how the tool fits the day.
launchd 7am briefing
Pipeline. Node and Claude-on-Azure synthesize vault state into a one-screen briefing every morning. Designed for skim, not depth. The model met me where I already start the day instead of asking me to come find it.
03 · What it taught me

Four principles for designing AI-native product surfaces.

These aren't laws. They're the patterns I keep returning to when I'm designing the AI-native tools I ship to colleagues at Bayer — and the ones I'd want to design at Anthropic.

01

Custom surfaces should be cheap to author.

The reason I built /bod and /eod is that Claude Code made them a twenty-minute job, not a two-week feature request. The threshold to extend the tool sat below the threshold to ignore the friction. That's a design decision, not a side effect.

Every AI tool should make its own extension surface this cheap. The product that lowers the cost of authoring its own primitives wins disproportionately, because the user starts shipping the features the team would never have prioritized.

02

The CLI is a design canvas, not just a panel.

Most "AI features" assume the chat window is the surface. Claude Code refuses that assumption. Hooks, skills, subagents, MCPs, slash commands — each is a designed primitive with its own shape, its own grammar, its own authoring story.

The design problem isn't "what goes in the chat." It's "what primitives compose into a working environment." That reframing is what makes Claude Code feel different from every other AI tool I use, and it's the reframing I'd want to keep extending.

03

Memory + skills + hooks compose into a personal OS.

I don't have a separate productivity app, calendar assistant, vault search, or briefing tool. Claude Code holds all of it because the primitives composed. Memory remembers me. Skills package the verbs I use most. Hooks decide when the environment speaks first. Together they're an operating system, not a feature set.

That composability is the design opportunity. The product that turns "an AI you use" into "an AI that holds your day" is doing the harder, more durable work.

04

The best AI products make their users more capable, not more dependent.

A year ago I couldn't ship working software. Now I ship AI-native tools to a hundred and fifteen colleagues. Not because I became a better engineer — because Claude is a capability-building partner. It teaches as it works. It defers when I should be the one deciding. It leaves me sharper than it found me.

That's the bar I'd want to design against. A surface that flatters the user's dependency on it is easy to build. A surface that grows the user's range is the one that earns trust at the scale Anthropic is operating at.

04 · What I'd want to design

Three things I think about often.

AThe grammar of subagents.

Right now I author subagents in YAML frontmatter plus markdown. It works, and the constraint is honest — the authoring surface is a config file because subagents are still a new primitive and the team is keeping the door open. But it leaves a real design question on the table: what does subagent authoring look like when it's a first-class surface for non-engineers? A teammate of mine could describe the agent they want in a paragraph. The grammar of subagents is one of the most interesting open spaces in the product.

BHooks as a user-facing concept.

Hooks are powerful and almost invisible to most users. There's a design problem here: how do you surface hook composition so that customizing your Claude Code environment feels like writing a recipe, not editing JSON? Hooks are where Claude Code stops being a tool you use and starts being a tool you shape. Making them legible to a wider audience without making them less powerful is the kind of problem I'd want to spend a year on.

CThe "personal OS" pattern at scale.

What I've built for myself in Claude Code is a personal operating system. Does that scale to a team — a shared vault, shared skills, shared subagents — without losing the intimacy that makes the personal version work? My instinct is that the answer involves designing for permeable membranes between personal and shared, not flat-sharing everything. But I'd rather work that out with a team that's already thinking about it than write about it from the outside.

Anthropic's bet that AI should be reliable, interpretable, and steerable is the bet I've made in my own building. I'd rather make it at the source.
